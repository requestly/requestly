Objective
Rewrite the Auth Tab of the API Client to deliver a correct, stable implementation of the existing authentication experience. This covers auth configuration at request and collection level, inheritance behaviour, variable support in auth fields, and auth applied to GraphQL introspection.

Success Criteria
User can:

Select an auth type on a request

Configure all fields for the selected auth type

Send a request and have auth applied automatically to the correct header or query param without manually writing the Authorization header

Set auth at collection level and have all child requests inherit it

Override collection-level auth on an individual request

Explicitly set No Auth on a request to suppress inherited auth

Use {{variables}} in any auth field

Use a post-response script to write a token to an environment variable and reference it in an auth field

Feature Scope
Auth Tab — Request Level
Per-request auth configuration.

In Scope

Auth type selector:

No Auth

Inherit from Parent (default for requests inside a collection)

API Key

Bearer Token / JWT

Basic Auth

Auto-population of the correct header or query param at send time — user never manually writes Authorization header

Variable interpolation ({{variable}}) supported in all value fields

Out of Scope

Any new auth types

Auth Tab — Collection Level
In Scope

Same auth type selector and field set as request level

Auth set at collection level applies to all requests in the collection unless overridden

Sub-collections inherit from their parent collection

Auth values resolved and applied at send time, not stored per-request

Out of Scope

Partial inheritance (e.g. inherit type but override a single field value)

Cross-collection auth sharing

Auth Field Definitions
API Key

Key (name of header or query param)

Value

Add to: Header / Query Params selector

Bearer Token / JWT

Token value

Auto-formatted as Authorization: Bearer <token>

Basic Auth

Username

Password (masked)

Auto Base64-encoded into Authorization: Basic <credentials>

Inheritance Model
In Scope

Three states for request-level auth:

No Auth — sends nothing, explicitly suppresses parent auth

Inherit from Parent — uses nearest parent (sub-collection → collection) auth

Own auth type — fully overrides parent, uses request-level config

Inheritance resolved at send time through the collection hierarchy

Visual indicator showing which parent the auth is being inherited from

Requests default to Inherit from Parent, collections default to No Auth.

Out of Scope

Partial overrides

Variables in Auth Fields
In Scope

{{variable}} interpolation in all auth field values

Variable hover preview (resolved value shown on hover) inside auth fields

Variables resolved through standard precedence chain: request → collection → environment → global

Out of Scope

Secret variable masking in auth field display (tracked under Variable Management)

Auth and GraphQL
In Scope

Auth tab available on GraphQL requests — same type selector and fields as HTTP requests

Auth credentials automatically applied to schema introspection queries, not just user-initiated requests

Out of Scope

GraphQL-specific auth schemes

Export Behaviour
In Scope

Auth type and field key names exported with the collection

All auth field values stripped on export regardless of whether hardcoded or variable references

Out of Scope

Selective value export

Encrypted credential export

Token Automation via Scripts
In Scope

Post-response scripts can extract tokens from a response and write to environment variables via rq.environment.set()

Variables written by scripts can be referenced in auth fields via {{variable}} — e.g. Bearer Token field set to {{authToken}}

Collection-level auth with a variable token value means all child requests automatically use the updated token once the variable is set

Out of Scope

Built-in token refresh scheduling

Any scripting runtime changes (tracked under Runtime Core)
